## 3. 공통코드 운용 전략

> 목적
- 도메인 상관없이 모든 테이블에 걸쳐 약속하는 일련의 번호 혹은 체계
- 중복으로 나타나는 데이터를 정규형에 맞게 체계화, 번호화하여 일관성 확보 및 관리/확장성 모두 보장할 수 있다.

> 추가 속성 필요할 경우
- 할인율, 수수료율 등 상세 속성을 공통코드화 시킬 수 있다.
- column 추가 및 EAV(Entity-Attribute-Value).

- case 1 : 백엔드 쿼리로 일원화

| 장점                 | 단점                |
| ------------------ | ----------------- |
| 쿼리 1번 실행 (네트워크 1회) | 복잡한 쿼리 구조         |
| 네트워크 통신 최소화        | JOIN 비용 증가        |
| DB 단에서 최적화 여지 존재   | 테이블 I/O 비용 증가     |
|                    | 공통 코드 쿼리 중복 가능    |
|                    | 변경점 발생 시 전체 수정 필요 |

- case 2 : Application 쿼리 분리(전체 공통코드 내용 및 매핑: front/세부 데이터 : back)

| 장점                     | 단점              |
| ---------------------- | --------------- |
| SQL이 단순하고 명료           | 추가 쿼리 실행 필요     |
| 쿼리 분리를 통한 성능 저해 최소화 가능 | 네트워크 통신 증가      |
| 책임 분리 명확               | N+1 문제 발생 가능    |
| 유지보수 용이                | 추가 쿼리 실행 요구     |
|                        | 시스템 관점에서 복잡도 증가 |

> trade off
- JOIN 횟수 증가 및 이로 인한 쿼리 복잡도 증가
- 공통코드 추출을 위한 쿼리문 중복, 이로 인해 변경점 발생 시 모든 쿼리를 수정해야 함

> Ways
- 백엔드 쿼리는 공통코드만을 제공하여 Join을 제거하고, description은 공통코드 테이블을 별도 조회하여 추출(N+1)
- 필요한 공통코드 내역을 처음부터 모두 저장해두었다가, 쿼리 내용에 따라 자연스럽게 매핑(*front에서 자주 보이는 방법)
- 공통코드 목록을 미리 불러두되, 이를 DB가 아닌 메모리에 올려두어 네트워크 비용을 감소한다(캐싱).
  - DB와의 동기화 문제 발생, 공통코드 수정 시 즉시 캐시 갱신 요구.
  - 다만 분산 규모가 큰 환경일 경우 메모리 캐싱 보다는 Redis와 같은 중앙 매니징 캐싱을 활용할 것.
    - 이 경우 TTL 및 자체적인 갱신을 진행하도록 유도!

> With Enum. </br>
> **시작은 단순하게 Enum으로 시작, 필요 시 공통코드 테이블을 운용하도록 하자.**
- 공통코드가 비즈니스 로직에 중요한 요인으로 작용하거나, 데이터 변경 및 추가 시 공통코드를 입력해야 하는 경우
- 데이터의 일관성을 보장할 수 있는 가장 강력한 수단.
- Enum의 속성을 관리하는 번거로움보다 중장기적으로 유지보수 및 관리가 가지는 용이성이 훨씬 우세, Enum 사용 적극 권장.
  - 기본적으로 Enum은 속성값이 아닌 코드값만 관리, 속성은 공통코드 테이블에서 관리.
  - 다만, 표기용도로만 사용한다면 공통코드 테이블만 사용
  - 변경이 너무 많이 발생할 경우 공통코드 테이블만 활용하는 것도 전략.
- 동기화 전략 : CI/CD with test case or 서버 실행 시 검증, 반드시 필수.

## 4. 계층 구조 설계

- 2차원 평면 구조의 테이블의 데이터 계층 구조를 설계하고 표현하기 위한 다양한 방법(패턴)

> 인접리스트 모델
- 자식 노드가 부모 노드 데이터를 함께 가지고 있는 형태로, 자기참조 외래키를 보유하고 있는 구조.
- Self join에 의해 Self Referencing이 이루어지는 구조이다.
  - 직관적이면서, 참조 무결성 조건을 걸 수 있기에 일관성 보장이 가능하다.
  - 데이터 추가 및 수정이 비교적 간단한 형태이다.

> - CTE 및 재귀 쿼리를 통한 쿼리 체계화/계층화
  - 인접리스트 구조로 되어있는 데이터 구조에서, 계층 깊이가 가변적인 상황에서 유연하게 대응할 수 있도록 SQL 표준으로 제공하는 쿼리 형태.
  - CTE(하나의 테이블로 사용하는 결과 집합, 임시 View)에 재귀쿼리를 적용하여 활용.

```sql
WITH RECURSIVE employees AS (
    SELECT .. FROM EMPLOYEE employee
    
    UNION ALL
    
    SELECT * , e2.depth + 1 FROM EMPLOYEE employee e1
    JOIN employees e2 on e1.employee_id = e2.parent_id 
) 
SELECT * FROM employees;
```
> 폐쇄 테이블 모델(Closure Table Model)
- 인접 리스트 모델이 가지는 쿼리 복잡도 증가 및 조회 성능 하락을 보완하는 구조
- 기존 인접 리스트 모델에서의 직속 부모 참조 관계가 아닌, 모든 깊이의 자손을 참조하는 관계
  - 계층구조의 내역, 노드에 해당하는 정보와 계층 구조의 관계(path)에 해당하는 정보 분리 운용.
  - 관계도 먼저 추출(path/where path.ancestor_id) + 노드 정보 추출(path.descedant_id = node.id)

```sql
SELECT * 
FROM employee e
JOIN employee_path ep 
ON e.employee_id = ep.descendant_id = e.employee_id
--WHERE e.employee_id = 1
WHERE ep.ancestor_id = 1;
;
```

혹은

```sql
SELECT * 
FROM employee e
JOIN employee_path ep 
ON e.employee_id = ep.ancestor_id = e.employee_id
--WHERE e.employee_id = 1
WHERE ep.descedant_id = 5;
;
```

> 폐쇄 테이블 모델은 대규모 데이터에 대해서 적용하는 것이 바람직.
- 두개로 분리 운용되는 테이블, 경로 추가 및 삭제 시 규칙 정립이 필요하는 등 세부적인 데이터 관리가 매우 복잡하다. 
- 복잡한 만큼 조회 시 빠르고 안정적인 성능을 낼 수 있다. 
  - 쓰기보다는 조회가 매우 빈번하게 발생하고, 데이터 및 카테고리 등 분류 단계가 복잡하고 많은 대규모 데이터에 대해 적용 적합.

**실무에서는 인접리스트 모델 + CTE/재귀로 충분히 가능, 데이터가 압도적으로 방대해질 경우 폐쇄 테이블 모델 적용을 적극 고려해볼만하다.**

## 5. 데이터 변경 이력 설계

> 변경 이력을 통한 감사 대응, 역추적, 상태 변동의 근거 등 그 자체로 중요도가 높은 작업.
- 데이터 상태를 단순히 저장하는 것이 아닌, 상태 변경 이력을 관리하는 것은 필수적인 과정이다.

## 5-1. 변경을 추적하기 위한 단순 속성 추가 기반의 데이터 모델링

> 기본적으로, </br>
> 단순 정보성 데이터 혹은 원장 데이터 모델링 시, </br>
> 누가 언제 등록하였는가에 대한 정보, 누가 언제 수정하였는가에 대한 정보를 모두 추가하는 것이 좋다.

최소한 누가 언제 데이터를 변경하였는지는 기록해두는 것이 좋다.
- created/updated at, by

위의 변경이력관리의 세부적인 추적(왜/상태변경 이전의 값)을 위해 추가 운용이 필요하다.
- updated_at, by, change_type, change_reason

**Tip** : 감사 대응을 위해 변경을 요청한 IP까지 속성에 추가하면 대응에 수월하다(*고객 변경? 관리자 변경?).
- client_ip(for audit)

## 5-2. 이를 기반으로한 세부 변경 이력 관리 

> 데이터의 변경 요청 주체 및 사유와 더불어, </br>
> 그 이전의 상태값을 누적 관리하기 위한 세부 이력 관리 전략을 세우자.

그 이전의 상태값은 무엇인가?
- prv_status/prv_changed_at : 단순 이전의 상태값 및 상태값 변경 시점 속성의 추가
- 이전 상태에 대한 전체 스냅샷 관리
  - 기존 원장 테이블 그대로 활용 : Only INSERT
    - 원장에 history_id를 추가하여, 기존 데이터는 update(is_now = false) + 새로운 데이터 insert(is_now = true)
    - 원장에 history_id 추가하여 생성기반 관리 및 유효기간 속성을 통한 관리(valid_from/valid_to) 
  - 세부 로그성 테이블 추가 운용 : 원장 + 세부 내역(로그) 
    - 원장에는 최신 데이터만 반영 + 모든 이력을 별도로 저장하여 분리 운용
      > for 원장 데이터 -- 시점 조회가 거의 필요없고, 데이터가 쉴새없이 변경
      - created_at (해당 데이터의 최초 등록일시, 변하지 않는 속성) 및 history_created_at (이력 데이터의 등록 일시, 이력이 적재될때마다 반영)  
      - 최초 데이터 등록 시점부터 이력을 저장한다.
      > for 정보성, 관리성 데이터 -- 시점 조회가 필요하고, 데이터가 일정기간 동안 유지
      - 특정 시점에 대한 데이터 이력 대응 시 유효기간(valid_from, valid_to) 속성 추가 가능, 이 경우 history_created_at은 운용 필요 없음.
        - 다만, 유효기간 update 및 신규 데이터 insert 트랜잭션 도중 문제 발생 시에 대한 동기화 처리 고려 필요

이 외 특별한 이력 관리 방안
- 컬럼 단위 변경 로그(Field-Level Tracking)
  - 변경된 컬럼에 대해서만 이력 관리한다.
  - 다수의 변경 컬럼의 경우, Json 자체를 저장할 수 도 있음.
- 도메인 별 이력 테이블 관리가 아닌, 중앙 관리형 공통 이력 테이블의 운용
  - 컬럼 단위 변경 로그에 확장하여, 변경 유형 등을 관리하는 Json 기반의 관리가 구조적으로 유리.

# 이력 관리의 중요한 점

> 가장 중요한 것은 변경 이력 관리에 대해 규칙을 지정하고, 해당 규칙대로 이력 관리 로직에 그대로 반영하는 것이다.
- 비즈니스 이력은 이력 테이블에 스냅샷으로 관리(감사 대응 등)
- 디버깅 정보 및 에러 스택 트레이스 등 일반 로그들은 DB에 저장하지 말고, 파일로 저장하고 관리한다(ELK Stack)

## 6. SOFT DELETE

> SOFT DELETE의 필요성
- 물리적인 영구 삭제로 인한 데이터 복원의 어려움
- 연관 데이터 삭제에 대한 별도 협의 필요 및 외래키 제약으로 인한 구조적 문제 등 데이터 관리의 어려움
- 삭제 시점 파악 불가
- 법적 및 감사 대응(데이터 보관)

> Patterns
- is_deleted (삭제여부) 플래그 속성 추가
  - 삭제시점 및 이로 인한 특정 기간 이후의 데이터 삭제 정책 및 삭제 이력 보관 정책 적용 불가
- deleted_at (삭제시점) 속성 추가
  - 단순 삭제시점, NULL or NOT NULL
  - 두 연관 데이터의 조인 조건 반드시 확인(모든 데이터의 deleted_at IS NOT NULL)
    - view 활용(활성 혹은 삭제 데이터만 보여주는 view 별도 운용)
- deleted_at / is_deleted 속성 동시 운용
  - 한 원장 테이블에 두 삭제 속성을 동시에 운용, 별도 view 운용하지 않고 간단하게 삭제 데이터 이력 관리 등 가능

> HARD vs SOFT
- HARD DELETE
  - 데이터 적재량이 많고 이력 관리(복구)에 큰 의미가 없는 로그성 데이터
  - 임시데이터(비즈니스적으로 복구할 필요가 없는 데이터)
  - 법적인 요구사항(일정기간 이상 시 개인정보 폐기)에 의한 완전 삭제
- SOFT DELETE
  - 법적인 요구사항 혹은 실무 상황에 의해 복구가 빈번한 데이터
  - 법적으로 보관의무가 발생하는 데이터(회원정보 및 거래정보 등)
  - 연관 데이터가 많아 삭제 시 협의 및 규칙 제정이 복잡한 데이터(상품 및 주문 데이터 등)

> 추가 세부 전략
- 데이터 도메인에 따른 deleted_at 운용 및 삭제 속성(status)에 대한 전략적 선택
  - 주문/체결과 같은 생명주기 존재 vs 게시글 댓글/좋아요와 같이 존재유무에 중점, 복구 가능성만 고려해야 하는 경우
  - 생명주기, 즉 삭제여부를 통한 이분법적 기준 적용이 실무에서는 불가능한 상황이 다수 존재.
    - 상품 삭제와 상품 판매 중단은 다른 의미.
    - 주문 삭제와 주문 취소는 다른 의미.
  - 그 외의 경우는 SOFT DELETE로 운용.
- 데이터 연결 관계 및 세부적인 이력 관리(생명주기) 등에 기반한 HARD/SOFT DELETE 적용 전략 선택
  - 데이터 연결 관계가 없거나 제한적이고, 성능을 우선적으로 고려하며 이력 관리는 데이터 아카이빙을 통해 해야 한다면 HARD DELETE를
  - 데이터 연결 관계가 강하고, HARD DELETE가 제한적이며 이에 따른 상태 기반 생명주기 관리 및 세부적인 이력관리가 필요하다면 SOFT DELETE를
- 인덱스 설계 시 deleted_at, status 속성은 마지막, 후순위로.
  - SOFT DELETE 시 적용하는 deleted_at 속성은 변별력도 낮고, 동등조건으로의 적용성도 너무 낮다.
  - status의 경우 동등조건이기에 인덱스 설계 시 고려가 필요, 단순 deleted_at의 경우 후순위로 고려
- UNIQUE 제약 조건 회피 방안(SOFT DELETE 시 물리적으로 존재하는 데이터로 인해 UQ 제약 상황 발생 가능)
  - SOFT DELETE 시점에 TIMESTAMP와 같은 특정 문자열 등을 붙여 기존의 데이터 값을 변경
  - UQ 데이터와 deleted_at을 하나의 복합 UQ조건으로 구성(*DB에 따라 NULL 중복 허용하므로 위험)
